Flaky Network Playout System Architecture

Our solution achieves zero-loss playback over lossy, high-jitter network environments by implementing a single-threaded event loop architecture combined with a hybrid Forward Error Correction (FEC) and dynamic Negative Acknowledgement (NACK) reliability scheme.

System Level Design Choices

  • Single-Threaded Non-Blocking Event Loops: Both the sender and receiver utilize a single-threaded select() event loop. This guarantees 100% thread safety, eliminates lock contention overhead, and drastically reduces memory footprints.
  • Socket Draining Optimization: To bypass scheduling pre-emptions by the operating system kernel, the receiver drains all pending packets from the socket using a non-blocking recvfrom() loop before evaluating playout deadlines. This prevents bursts of misses due to scheduling delays.
  • 2-out-of-3 FEC Pattern: The sender piggybacks the payload of frame i-1 onto frame i for 2 out of every 3 frames. This ensures that any isolated packet loss is recovered instantly without wait time or RTT delays, maintaining an overhead of 1.69x (well below the 2.0x cap).
  • Adaptive NACK Retransmissions: NACKs are scaled dynamically: unprotected frames are NACKed aggressively (at min_delay + 15ms) to ensure prompt recovery, while protected frames are NACKed conservatively (at delay_ms * 0.75) to minimize bandwidth overhead.

Component Details

Sender (sender.c)

Maintains a ring buffer of the last 1024 frames. Each frame is indexed by seq % 1024. When a NACK is received via the feedback port 47004, the sender checks the ring buffer cache and retransmits the packet, rate-limiting retransmissions of the same packet to once every 10 ms to prevent feedback storms.

Receiver (receiver.c)

Stores incoming frames in a 2048-frame ring buffer to automatically handle packet reordering and duplicate discard. It runs a playout scheduler check every loop iteration and sends frames to the player exactly 2 ms before their strict deadline.
